# 我花了半年，给自己造了一个数字分身

## 起因

去年某天，我在回一封邮件的时候突然意识到一件事：

我不记得我们上次谈到什么结论了。

不是因为那件事不重要，而是因为过去三个月里，关于这个项目的信息散落在二十多个微信群、几十封邮件、还有几个本地 Word 文档里。我每次想拿到完整的上下文，都要花半小时去翻记录。

我当时的第一反应是：买个好用的笔记软件？

但我很快意识到，问题根本不在于笔记。问题在于：**信息本来就在那里，只是没有人帮我把它们连起来。**

我的微信里有几千条和合作方的聊天记录。我的邮件里有几百封和同事往来的邮件。我的硬盘里有几十个项目文件夹。这些东西加在一起，其实已经完整地记录了我数字生活的全貌——只是从来没有人去读它、理解它、帮我提炼出来。

于是我开始想：能不能造一个东西，专门干这件事？

---

## 数字分身是什么意思

我知道"数字分身"这个词听起来有点科幻。让我解释一下我的理解。

现在大家用的 AI 助手，不管是 ChatGPT 还是各种国产大模型，有一个共同的特点：**它们是无状态的**。每次对话都从零开始，它不认识你，不知道你在做什么项目，不知道你的人际关系，不知道你上周做了什么决定。

你每次用它，都要重新介绍背景。

这就好比你每次开会，都得先跟所有人重新自我介绍一遍，然后从头解释项目背景，然后才能进入正题。这不是效率问题，这是根本性的设计缺陷。

**我想要的不是这样一个工具。我想要的是一个已经认识我的存在。**

它知道张总是个拖延症晚期患者，但只要他承诺了就一定会兑现。
它知道 A 项目卡在了甲方审批环节，B 项目的技术路线上周刚改了方向。
它知道我最近在关注哪个研究方向，上个月看了哪些论文。
它知道我今天应该先处理哪三件事。

这就是我说的数字分身——**一个关于你的持久存在的模型，由你自己的数据构建，为你一个人服务。**

---

## 我做了什么

我把这个东西叫做 Aegis，用 Python 写的，跑在自己的 Windows 电脑上。

它做的事情说起来其实不复杂：

**第一，持续读取数据。**

每15分钟增量同步一次微信聊天记录（解密本地数据库，只处理有变化的部分）。每30分钟检查一次邮件（163和Gmail都支持）。本地文件实时索引，支持 .md .docx .pdf .py 等格式。学术RSS每天早上自动拉取。

所有数据都在本地处理，不上传到任何云端。

**第二，从数据里提炼记忆。**

这是最核心的部分。原始数据（聊天记录、邮件）是噪音，真正有价值的是从里面提炼出来的东西：

- 这个人是谁？他和我什么关系？最近在聊什么？重要程度如何？
- 这个项目现在什么状态？上一次的决定是什么？
- 我现在最需要处理的事是什么？

这些被组织成四层记忆，全部以 Markdown 文件存储，放在一个独立的 git 仓库里：

```
focus.md        — 当前最需要处理的事（自动维护，≤15条）
people.md       — 重要联系人索引，按重要度排序
projects/*.md   — 每个项目的独立档案，记录决策和背景
知识库          — FTS5全文索引 + ChromaDB向量搜索
```

**第三，主动向你汇报。**

每天早上8点，它会把邮件、微信、学术资讯融合成一份简报发到我邮箱。规则引擎先打分，质量不够才调用大模型重写——这样做是为了省token，毕竟大部分时候规则已经够用了。

**第四，响应你的指令。**

给自己发一封邮件，主题写 `Aegis: 搜索 国自然 截止日期`，它找到结果之后会邮件回复你。在微信里发 `Aegis: 今天有什么需要处理的`，它会把焦点清单发回来。

这样做的好处是：不管我在哪，用手机就能查询和控制。

## 一些有意思的技术细节

**微信解密**是整个系统里最"硬核"的部分。微信的本地数据库用 SQLCipher 4 加密，密钥跟硬件绑定。解密之后是标准 SQLite，可以直接查询。我用的是增量同步方案——用 mtime 跟踪每个数据库文件的修改时间，只解密有变化的部分，避免每次全量处理。

**记忆写入的安全性**是我花时间最多的地方。所有写入都经过统一入口，线程锁保护，每次写入记录到 SQLite，然后 git commit。这意味着任何一次写入都可以回滚：

```bash
git -C data/memory log --oneline
git -C data/memory revert <hash>
```

人类可读的 Markdown + git 版本控制，让这套记忆系统的生命周期远超任何 app。

**去重**是一个被低估的问题。同一件事可能从邮件、微信、RSS 三个渠道都被提取到焦点列表里。我用的是字符级 bigram Jaccard 相似度，专门为中文优化——因为中文不能简单地按空格分词。

**token 节省**也是设计重点。规则引擎先对简报质量打分（0 token），只有分数低于7分才触发大模型重写。实际使用下来，大约70%的日报都是规则引擎直接生成的，不消耗任何API额度。

## 用下来是什么感受

用了半年，几个真实的体会：

最明显的变化是**我不再害怕"失忆"了**。以前出差回来，要花半天时间重新同步状态。现在打开邮件看看简报，五分钟就清楚了昨天、前天发生了什么，谁需要跟进，什么事情到期了。

另一个变化是**搜索方式变了**。以前我找一个东西，要先想想"这是邮件还是微信还是文档"，然后分别去搜。现在直接发指令，它把所有来源的结果合在一起返回，还带相关度评分和摘要。

当然也有还没解决好的地方。提炼出来的记忆准确率不是100%，偶尔会有错误的归属或者理解偏差，需要手动纠正。微信的解密也偶尔会因为微信版本更新而需要重新适配。

## 为什么要做成本地的

这是个主动选择，不是技术限制。

市面上有一些类似定位的产品，但都有一个共同点：你的数据要上传到他们的服务器。他们帮你存储、帮你分析、帮你同步。

我不接受这一点。

不是因为偏执，而是因为：**个人关系网络、工作决策记录、私人通讯内容，这些东西的价值恰恰来自于它们的私密性。** 一旦上云，你就不再是这些数据唯一的拥有者了。

Aegis 的所有数据都在本地。大模型调用只发送文本摘要，不发送原始聊天记录或邮件全文。关掉网络，除了 AI 功能，其他一切正常运转。

这是我认为个人 AI 助理应该有的样子。

## 开源了

代码放在 GitHub：**https://github.com/ZhixiangWang-CN/aegis**

技术栈：Python 3.11 + FastAPI + SQLite + ChromaDB + APScheduler，大模型接的是火山引擎 Doubao（OpenAI 兼容接口，换成其他接口也很容易）。

现在还在迭代中。下一步想做的是接 Ollama，实现完全离线运行；以及 WhatsApp 接入，给海外用户也用上。

如果你也在被"信息散落到处都是但就是找不到"这个问题困扰，可以看看。
